# 8. 장애 대응 및 운영 안정성 설계

## 1. 개요
운영 전담 인력이 없고 5명의 개발자만 있는 상황이므로, "장애가 발생하지 않는 시스템"보다는 **"장애가 발생해도 개발자가 자다가 일어날 필요 없이 시스템이 알아서 복구(Self-Healing)하고, 최소한의 알림만 주는 환경"**을 만드는 데 집중했습니다.

## 2. 장애 시나리오별 대응 전략

### 2.1 트래픽 폭주 (Traffic Spike)
- **상황**: 특정 시점에 감성 분석 요청이 평소의 3~4배 이상 폭주함.
- **대응 (사전 차단)**: 
  - K8s의 **HPA (Horizontal Pod Autoscaler)**를 통해 평균 CPU 리소스 사용량이 70%를 넘기면 파드를 최대 4개까지 자동으로 스케일아웃합니다.
  - FastAPI 단에서 들어오는 요청 레이턴시가 500ms 이상으로 지연되기 전에 파드가 선제적으로 증설됩니다.
- **효과**: 개발팀이 수작업으로 스펙업(Scale-up)을 해줄 필요 없이 유연하게 처리량(Throughput)을 확보합니다.

### 2.2 파드(컨테이너) 앱 크래시
- **상황**: 메모리 누수(OOM)나 파이썬 런타임 에러로 Uvicorn 프로세스가 비정상 종료됨.
- **대응 (Self-Healing)**:
  - Docker 레벨에서는 `restart: always`를 걸었고, K8s 환경에서는 `Liveness Probe`가 10초마다 `/health` 엔드포인트를 찌릅니다. 응답이 없다면 K8s Control Plane이 해당 파드를 즉각 재시작(Restart)합니다. 
  - 파드는 기본적으로 2개(`replicas: 2`)가 돌아가고 있으므로 한쪽이 재시작되는 동안 다른 쪽이 트래픽을 온전히 커버하여 서비스 중단(Downtime)을 막습니다.

### 2.3 새 모델 배포 중 장애 (Bad Release)
- **상황**: AI 팀에서 준 새 모델 가중치가 깨져서 서버가 안 뜨거나 계속 에러를 뱉음.
- **대응 (안전망)**:
  - CI 파이프라인에서 1차적으로 `pytest`가 실패하여 배포를 막습니다.
  - 만약 테스트를 뚫고 이미지가 K8s로 넘어오더라도, `Readiness Probe`가 `/health` 접근을 확인하기 전까진 트래픽을 새 파드로 흘려보내지 않습니다(Rolling Update). 즉, 서버가 정상적으로 뜰 때까지 기다렸다가 구버전과 스위칭하므로 "배포 쳤더니 서비스 즉시 먹통"이 되는 사태를 원천 차단합니다.

## 3. 안정성 확보를 위한 리소스 격리

- **Resource Limits**: (limits.cpu: 1000m, limits.memory: 1Gi)
  한 파드가 무한 텍스트 루프나 모델 오류로 호스트 서버의 메모리를 다 갉아먹는 상황을 막기 위해 락(Lock)을 걸었습니다. 파드 하나가 죽더라도 물리 노드나 다른 앱 파드는 안전하게 격리됩니다. 

## 4. 운영 알림 체계 (계획)
현재는 모니터링 툴(Grafana)에서 지표만 보고 있지만, 향후 Prometheus Alertmanager를 붙여서 "5xx 에러가 1분에 10번 이상 발생" 같은 크리티컬한 이슈에만 팀 슬랙으로 알람이 오게끔 구축할 계획입니다. (자잘한 재시작은 K8s가 조용히 알아서 처리하도록 둡니다.)
