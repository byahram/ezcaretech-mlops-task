# 9. Observability 설계 (메트릭, 로그, 트레이스)

## 1. 개요
운영자가 없는 개발팀 환경에서는 장애 파악 속도가 생명입니다. 시스템의 내부 상태를 외부에서 한눈에 파악할 수 있는 Observability(관측성)를 확보하기 위해 3가지 기둥(Metrics, Logs, Tracing)을 어떻게 설계했는지 정리했습니다.

## 2. 메트릭 (Metrics)

- **도구**: Prometheus
- **설계 의도**: 서버가 얼마나 바쁜지, 메모리는 얼마나 먹고 있는지 수치로 확인합니다.
- **수집 지표**:
  - **애플리케이션 지표**: `/metrics` 엔드포인트를 열어서 평균 레이턴시, 초당 요청 수(RPS)를 수집합니다.
  - **인프라 지표**: K8s 환경이므로, 컨테이너별 CPU/Memory 사용량을 주기적으로 가져옵니다. 
- **활용**: HPA 스케일링의 기준점이 되며, 레이턴시가 500ms를 초과하거나 5xx 에러율이 1%를 넘어가면 AlertManager를 통해 즉시 슬랙 알림을 쏘도록 기준을 잡았습니다.

## 3. 로그 (Logs)

- **도구**: Loki + Promtail
- **설계 의도**: 메트릭을 보고 "문제가 생겼네?"를 알았다면, "왜 문제가 생겼지?"를 찾기 위한 단서입니다.
- **수집 방식**:
  - `python-json-logger`를 써서 FastAPI의 모든 로그(추론 결과, 에러 메시지)를 JSON 포맷으로 표준 출력(Stdout)합니다.
  - 컨테이너 바깥에 붙은 Promtail이 이 표준 출력을 긁어서(DaemonSet 방식) Loki 서버로 던집니다.
- **활용**: Grafana에서 메트릭 그래프(예: 에러 스파이크)를 드래그하면, 그 시간대에 찍힌 JSON 로그들만 필터링해서 바로 볼 수 있어 원인 분석 속도가 획기적으로 줄어듭니다.

## 4. 분산 트레이스 (Traces) - *Future Work*

- **현 상황**: 현재는 단일 API 서버(1-Tier)로 구성되어 있어 분산 트레이싱까지는 과도한 오버헤드라고 판단해 제외했습니다.
- **도입 계획**: 향후 모델 서빙 앞단에 `API Gateway`나 `BFF(Backend for Frontend)` 서버가 추가되어 마이크로서비스(MSA) 구조로 넘어간다면, **OpenTelemetry (또는 Jaeger)**를 도입할 예정입니다. 
- **효과**: 트랜잭션마다 고유 Request ID를 부여해 "Gateway -> Auth -> ML API" 중 어느 구간에서 병목(지연)이 발생하는지 정확히 짚어낼 수 있습니다.

## 5. 설계 결론 및 트레이드오프

- 외부 SaaS(DataDog, NewRelic 등)를 쓰면 이 모든 게 한 번에 붙지만, 사내 비용 정책("지정된 가용 자원만 사용")에 맞춰 오픈소스(Prometheus, Loki) 스택을 사내 인프라에 직접 올리기로 타협했습니다.
- 설정의 번거로움이라는 비용을 지불하는 대신, 완벽한 데이터 주권 유지와 라이선스 비용 절감이라는 트레이드오프를 선택한 셈입니다.
