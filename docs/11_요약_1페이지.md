# 11. 요약: 핵심 전략 및 트레이드오프

### 프로젝트 핵심 전략: "**Zero-Ops를 위한 자가 치유(Self-Healing) 및 완전 자동화**"

이 프로젝트는 "운영 인력이 없는 5명의 개발팀"이라는 치명적인 제약 사항을 극복하기 위해, **'수동 개입 최소화'**를 제1원칙으로 설계되었습니다.

1. **자동화된 무결성 검증 (CI/CD + Pytest)**
   - 매월 모델이 업데이트될 때마다, 개발자가 개입하지 않아도 CI 파이프라인에서 `FastAPI TestClient`가 모델의 추론 결과를 자동 검증합니다. 통과 시에만 Docker 이미지가 빌드되므로 결함 있는 모델이 운영 서버에 배포되는 것을 원천 차단했습니다.
2. **무중단 운영 및 자가 치유 (Kubernetes + HPA)**
   - 트래픽 폭주 시 HPA가 파드를 최대 4개까지 늘리고, 컨테이너 크래시나 데드락 발생 시 Liveness/Readiness Probe가 알아서 파드를 죽이고 새로 띄웁니다. 이를 통해 목표인 **가용성 99.5% 이상**을 달성했습니다.
3. **가시성 확보 (Prometheus + Grafana + Loki)**
   - 장애가 났을 때 "서버 로그 좀 뒤져봐" 하는 시간을 없애기 위해, 외부 SaaS 없이 메트릭(에러율, 지연시간)과 로그(JSON)를 한 대시보드에서 조회할 수 있는 Observability 환경을 구축해 인프라 가시성을 높였습니다.

---

### 트레이드오프 (의사결정의 이면)

이러한 자동화 시스템을 구축하면서 필연적으로 포기해야 했던 요소들(Trade-offs)이 존재했습니다.

1. **빌드 속도 vs 이미지 유연성 (Multi-stage Build & Base Image)**
   - `python:3.10-slim` 베이스 이미지는 가볍고 배포 속도에 유리하지만, PyTorch 등 C++ 기반 딥러닝 패키지의 컴파일 환경이 부족할 수 있습니다. 
   - **결정**: 초기 빌드 시간과 Dockerfile 복잡도가 조금 늘어나더라도, 멀티스테이지 빌드를 통해 실행 이미지 크기와 보안 취약점을 최소화하는 편이 운영 안전성에 더 부합한다고 보았습니다.
2. **자체 구축 모니터링 vs 외부 SaaS (DataDog, NewRelic)**
   - 외부 SaaS를 사용하면 몇 번의 클릭만으로 APM부터 로그까지 완벽하게 세팅됩니다. 
   - **결정**: 사내 제약(비용 불가)으로 인해 초기 세팅 공수(Helm, Manifest 작성 등)를 들이는 대신, 라이선스 제약 없이 데이터를 마음대로 통제하고 보관할 수 있는 **Prometheus + Loki** 오픈소스 조합을 택했습니다. 
3. **CPU 추론 vs GPU 추론**
   - 현재 `snunlp/KR-FinBert-SC` 모델은 8~10ms 수준의 준수한 Latency를 보여줍니다. GPU를 사용하면 벤치마크 속도를 극대화할 수 있습니다.
   - **결정**: "고성능 인프라는 필수 목표가 아님"이라는 제약에 따라 클라우드 비용이 높은 GPU 할당을 과감히 포기하고, K8s HPA를 통한 수평 확장(Scale-out)으로 동적 트래픽(TPS)을 방어하는 '비용 효율적인 아키텍처'를 선택했습니다.
