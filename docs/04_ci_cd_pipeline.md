# 4. CI/CD 파이프라인 설계 (설정 파일 + 설명)

## 1. 개요 및 파이프라인 흐름
혼자서 이걸 다 관리하고 배포하려다 보니 완전 자동화가 필수였습니다. 그래서 익숙한 GitHub Actions를 기준으로 CI/CD를 짰습니다.

**전체 흐름**:
1. `main` 브랜치에 코드가 푸시되면 워크플로우 자동 실행
2. 파이프라인 환경 셋업 및 패키지 설치
3. `pytest`로 테스트 코드 실행 (실패하면 여기서 빌드 중단)
4. 테스트를 통과하면 K8s나 서버에 띄울 Docker Image 빌드

## 2. YAML 설정 파일 설명

CI 파이프라인을 `.github/workflows/ci-cd.yml`에 설정해뒀습니다. 

- **Test Step**: 파이썬 `3.10` 환경을 띄우고 `requirements.txt`에 있는 라이브러리(`fastapi`, `pytest` 등)를 깐 다음 테스트를 돌립니다. 여기서 테스트가 깨진 채로 릴리즈 이미지부터 만들어지는 사고를 막았습니다.
- **Build/Push Step**: 테스트를 통과해야만 Docker 이미지가 만들어집니다. `docker/build-push-action`을 썼고, 빌드할 때 매번 Pytorch 같은 무거운 걸 처음부터 다운받으면 너무 오래 걸리니까, GitHub Actions 전용 캐시(`type=gha`)를 설정해서 빌드 속도를 챙겼습니다.

## 3. 회고 및 기대 효과

처음에 테스트를 빌드 안에다 넣을까 고민했는데, CI 단계에서 로직 결함(예: 모델 로딩 실패나 HTTP 500 에러)을 먼저 걸러내는 게 나중에 운영 서버 롤백하는 것보다 훨씬 맘 편하다고 판단했습니다. 

한 번 이렇게 구성해두면 월 1~2회 AI 팀이 새 모델 줬을 때 그냥 푸시만 띡 하면 되니까, 인프라 운영 스트레스가 거의 없을 것 같습니다.
