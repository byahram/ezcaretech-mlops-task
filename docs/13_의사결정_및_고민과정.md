이 문서는 "5명의 개발자로 구성된 서비스 팀의 유일한 MLOps 담당자"라는 가상 시나리오 하에, 본 과제를 수행하면서 직면한 주요 문제들과 그 해결 과정, 그리고 핵심 의사결정의 배경을 구체적으로 정리한 내용입니다.

## 1. FastAPI `TestClient`와 `httpx` 버전 충돌 이슈 해결

**[상황 및 문제 인식]**

CI/CD 및 자동화된 테스트 파이프라인(Pytest) 구축 과정에서 아무 이상이 없어 보이던 테스트 코드가 수집(collection) 단계에서부터 아래와 같은 에러를 발생시켰습니다.
`TypeError: Client.__init__() got an unexpected keyword argument 'app'`

**[원인 분석]**

에러 스택 트레이스를 추적해본 결과, 최신 `FastAPI(0.109.0)`의 내부 의존성인 `starlette` 패키지가 최신 버전의 `httpx (>=0.28.0)` 모듈과 호환되지 않아 발생하는 문제임을 파악했습니다.

**[해결 과정 및 의사결정]**

1. 테스트 코드 자체(Mocking 적용 여부 등)를 수정할 수도 있었으나, 근본적인 원인은 라이브러리 간의 버전 충돌이었습니다.
2. 따라서 운영 코드에 영향을 주지 않고 테스트 환경의 일관성을 유지하기 위해, `api/requirements.txt` 내 테스트용 의존성을 `httpx<0.28.0`으로 핀(Pinning 하향 고정)하여 의존성 충돌을 해결했습니다.
3. **결과**: 수정 후 `pytest tests/` 명령이 정상적으로 2개의 테스트(health check, predict)를 약 8ms 이내에 통과하는 것을 확인했으며, 이 룰을 CI 파이프라인에도 적용해 안정적인 배포 라인을 구축했습니다.

## 2. 모니터링 컴포넌트 간 Docker Context 참조 오류

**[상황 및 문제 인식]**

`monitoring/docker-compose.yml`을 작성하여 로컬 테스트를 돌리려던 중, 프로젝트 루트에 있는 애플리케이션 `Dockerfile`을 찾지 못해 빌드가 실패하는 문제가 발생했습니다.

**[원인 분석]**

Docker Compose가 명령어를 실행한 경로(`monitoring/`)를 기준으로 Build Context를 한정해 버리기 때문에, 그 상위 디렉터리(`../`)에 위치한 파일들에 접근할 권한이나 경로 매핑이 없는 것이 원인이었습니다.

**[해결 과정 및 의사결정]**

1. 모니터링용 스택이 애플리케이션 코드를 관할할 수 있도록 `docker-compose.yml` 내부 설정 파일의 `build: context`를 `..`(상위 폴더)로 변경했습니다.
2. 또한 `dockerfile` 경로는 `Dockerfile`로 직접 명시하여, 하위 디렉터리에서 실행하더라도 프로젝트 전체 루트를 빌드 컨텍스트로 묶어 컨테이너 다중 관리를 용이하게 설정했습니다.
3. 이 경험을 통해 마이크로서비스 확장을 염두에 둘 때, 환경 구성원(App, Prometheus, Grafana 등) 간 디렉터리 역할 분리 및 의존성 경로 관리의 중요성을 다시 한번 실감했습니다.

## 3. 고성능 인프라 배제 상황에서의 트래픽 방어 전략 (GPU vs CPU)

**[상황 및 문제 인식]**

사용할 모델은 `snunlp/KR-FinBert-SC`입니다. GPU 기반 추론이 CPU 기반 추론 대비 월등한 처리량(TPS)과 지연시간 단축을 가져올 수 있으나, 과제 제약 조건상 "고성능 인프라는 필수 목표가 아님(유상 자원 사용 지양)"이 명시되어 있었습니다.

**[원인 분석 및 고민]**

무작정 GPU를 도입하기엔 인프라 비용 부담이 커지고 MLOps의 복잡도(CUDA 드라이버 관리, GPU 프로비저닝 등)가 올라갑니다. 반면 오직 CPU 1대만 운영할 경우, 트래픽 유입 스파이크 시 99.5% 가용성을 위협할 수 있습니다.

**[해결 과정 및 의사결정]**

1. **아키텍처 스케일아웃(Scale-out) 중심 전환**: 
   - 개별 컨테이너 스펙을 올리는 Scale-up (GPU 도입) 방식 대신, K8s 환경에서 CPU 파드 개수를 유동적으로 늘리는 **Scale-out (HPA 도입)** 전략을 채택했습니다.
2. **K8s HPA (Horizontal Pod Autoscaler) 세팅**:
   - `cpu averageUtilization: 70`으로 설정해, 특정 순간에 트래픽이 몰려 CPU 연산(추론 부하)이 70%를 넘으면 즉각적으로 파드가 2개에서 최대 4개까지 복제되도록 설정했습니다.
3. **효과 및 근거**: 
   - 결과적으로 비용 효율성을 잡으면서도 "월 1~2회 새 모델 배포"나 "갑작스러운 API 부하"에 유연하게 대응할 수 있는 Zero-Ops 환경을 달성했습니다.

## 4. 운영 인력 부재에 따른 "Self-Healing 시스템" 지향

**[상황 및 문제 인식]**

가상의 서비스 팀은 저를 제외하면 백엔드/프론트 개발자 5명뿐입니다. 장애가 발생했을 때 인프라(서버)에 접근해서 일일이 포트를 열고 재시작을 해줄 전담자가 부족합니다.

**[해결 과정 및 의사결정]**

1. 철저하게 **관측성(Observability)**과 **자가 복구(Self-Healing)** 쪽으로 시스템 코어를 맞췄습니다.
2. `Liveness/Readiness Probe`에 짧은 주기의 헬스체크(`/health`)를 달아, 어떤 파이썬 예외 상황이나 OOM 크래시가 발생하더라도 즉각적으로 컨테이너를 K8s가 자동 재부팅하도록 세팅했습니다.
3. 에러 분석 시에도 서버에 들어가지 않고, Grafana 모니터링 화면 하나에서 Prometheus의 메트릭 스파이크 시간대와 Loki의 애플리케이션 JSON 텍스트 로그를 맵핑해서 볼 수 있는 구조를 구축하여 트러블슈팅 소요 시간을 획기적으로 줄였습니다.
