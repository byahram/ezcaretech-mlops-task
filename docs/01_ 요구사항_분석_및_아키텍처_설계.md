# 1. 요구사항 분석 및 아키텍처 설계

## 1. 요구사항 요약

기존에 AI 팀에서 월 1~2회 업데이트해서 주는 감성 분석 모델을, 별도의 인프라/운영 전담자 없이 5명의 개발자(저 포함)가 안정적으로 서비스할 수 있게 만드는 과제입니다.

- **주요 제약사항**: 전담 운영자 없음, 완전 자동화 필수, 가용성 99.5% 달성
- **모델 정보**: `snunlp/KR-FinBert-SC` (CPU 추론 허용, 고비용 GPU 불필요)

## 2. 아키텍처 설계 및 기술 스택 선정

운영 리소스를 최소화하면서 트래픽을 감당하기 위해 익숙하고 검증된 스택 위주로 골랐습니다.

- **Serving API**: **FastAPI**
  - 비동기 처리가 쉬워서 성능 뽑기가 좋고, Pydantic 기반으로 데이터 검증이 편합니다. (Swagger 문서 자동 생성되는 점도 팀 협업에 큰 장점입니다.)
- **Containerization**: **Docker & Docker Compose** (로컬/단일 서버 환경)
  - Ubuntu 22.04 환경의 의존성을 분리하기 위해 `python:3.10-slim`으로 경량화했습니다.
- **Orchestration**: **Kubernetes**
  - 무중단 배포(Rolling Update)와 장애 발생 시 자동 재시작(Self-healing)을 위해 K8s 매니페스트(Deployment, Service 등)를 구성했습니다.
- **Observability**: **Prometheus + Grafana + Loki**
  - 외부 SaaS를 못 쓰기 때문에, 가장 레퍼런스가 많고 구축이 편한 조합으로 로깅 및 메트릭 수집 파이프라인을 짰습니다.

## 3. 정량적 분석 및 처리량(TPS) 산정

현재 개인 랩탑(MacBook Pro, M 시리즈 MPS 활용) 기준으로 로드 테스트를 돌려봤습니다.

- **평균 추론 Latency**: 약 8~10ms 수준
- **예상 최대 TPS**: 단일 컨테이너 기준 약 100~120 req/s

실제 프로덕션에서는 CPU 추론만 수행하므로 이것보단 Latency가 다소 늘어날 수 있습니다. 하지만 가용성 99.5%에 맞추기 위해 Liveness Probe를 연동하고, 파드 개수를 최소 2개(replicas: 2)로 유지하게 K8s를 설정할 계획입니다. 이렇게 하면 한 쪽이 죽거나 업데이트되는 동안에도 서비스 중단이 없습니다.
